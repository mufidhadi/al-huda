# Development Log: Tanya Quran Hadist

**Project Name:** Tanya Quran Hadist  
**Date:** 25 Maret 2026  
**Developer:** Gemini CLI & Mas Mufid  

## 🚀 Project Overview
Membangun sistem kueri dan dataset terpadu untuk teks suci Al-Quran dan Hadits. Fokus utama adalah pada integritas data, pemisahan struktural (Sanad vs Matan), dan optimasi performa penyerapan data menggunakan infrastruktur modern. Kini telah mendukung kueri semantik lintas sumber dengan library mesin pencari terpadu yang telah diaudit performanya, divalidasi dengan pengujian otomatis skala besar, dan memiliki dokumentasi API standar Postman.

## 🛠️ Tech Stack
- **Python 3.12**: Bahasa pemrograman inti.
- **UV**: Manajer paket dan proyek tercepat.
- **FastAPI & Uvicorn**: Framework API modern untuk performa tinggi.
- **PostgreSQL (v16) + pgvector**: Basis data relasional dengan kapabilitas kueri semantik.
- **Sentence-Transformers (E5-Small)**: Model lokal untuk pemrosesan teks menjadi vektor 384 dimensi.
- **Pillow**: Library rendering gambar untuk fitur social sharing.
- **pytest**: Framework pengujian otomatis.
- **psutil**: Monitoring resource sistem.
- **tqdm**: Visualisasi progres proses intensif.

## 📝 Experience & Task List

| Task | Issue | Tried Approach | Result | Lesson Learned (Project) | Lesson Learned (General Dev) | Lesson Learned (AI Prompting) |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **Quran Ingestion** | Duplikasi data karena proses paralel tumpang tindih. | TRUNCATE sederhana. | Data terduplikasi (11k ayat). | Domain religi sangat sensitif terhadap jumlah baris (6236 ayat). | Selalu lakukan *cleanup* proses background (`taskkill`) sebelum operasi ETL. | AI butuh instruksi pembersihan proses eksplisit saat menggunakan mode background. |
| **Gap Analysis & Resume** | Koneksi terputus di tengah proses besar. | Pengecekan status DB per baris. | Ingestion cerdas (resume). | Dataset Quran butuh validasi jumlah ayat per surah (State-awareness). | Implementasikan *Idempotency* pada setiap fungsi penulisan data. | Berikan AI konteks arsitektur "Smart Resume" agar tidak menulis ulang dari nol. |
| **Indexing Performance** | INDEX lambat pada tabel gemuk. | Indexing setelah isi data. | `LockNotAvailable` error. | Indeks pada `nomor_surah` mempercepat kueri audit. | Gunakan `VACUUM ANALYZE` sebelum melakukan indexing pada tabel besar. | Gunakan skrip Python terpisah untuk SQL kompleks agar terhindar dari isu karakter shell. |
| **Hadith Master Data** | Struktur data dari API tercampur-campur. | Pembuatan tabel master Imam, Kitab, & Derajat. | Integritas referensial terjaga. | Normalisasi data master di awal mempermudah kueri kompleks (Join). | Integritas referensial terjaga melalui *Foreign Key Constraints*. | Berikan AI tugas pembuatan skema master secara terpisah untuk menjaga fokus. |
| **Sanad-Matan Split** | Tidak ada API yang memisahkan Sanad & Matan. | Algoritma *Splitter* lokal (V1-V2). | Akurasi awal belum stabil. | Sanad dan Matan memiliki pola transisi linguistik yang dinamis di Indonesia. | *Domain-specific parsing* sederhana seringkali lebih efektif daripada menunggu API sempurna. | Gunakan sampling audit untuk menguji limitasi logika AI secara forensik. |
| **Transactional Safety** | Duplikasi hadits saat range paralel gagal. | Transactional block (commit/rollback). | Data tetap bersih. | Integritas Hadits bergantung pada atomisitas per rentang (*range*). | Atomisitas adalah harga mati dalam sinkronisasi data paralel. | Berikan AI referensi kode yang sukses sebelumnya untuk meniru pola transaksi. |
| **Parsing Logic V3** | Kasus khusus (ID 4, 5, dsb) gagal dipisah. | Implementasi *Rightmost Search* (`rfind`) untuk mencari perawi terakhir. | Akurasi pemisahan >95% untuk Bukhari. | Struktur hadits sangat dinamis; posisi tanda baca terakhir adalah kunci pemisah terbaik. | Pendekatan *Positional Split* (rsplit) lebih tangguh daripada *Keyword Match* statis. | Gunakan sampling audit forensik (IDs 4, 5, dsb) untuk menguji limitasi logika. |
| **Vector Embedding** | Butuh pencarian makna (Semantic Search). | Local E5-Small model + pgvector. | 100% Independent AI infra. | E5-Small (384 dim) adalah model optimal untuk pencarian asimetris Quran/Hadits. | Shared Primary Key (1-to-1) pada tabel embedding meningkatkan performa JOIN. | Jelaskan konsep "Independence" model (cached) agar pengguna merasa aman. |
| **Unified Search Engine** | Perlunya logika pencarian konsisten untuk UI. | Library `QuranHadithSearch` dengan perankingan Cosine Distance. | Pencarian lintas sumber berhasil dengan format output seragam. | Abstraksi logika ke dalam library memudahkan skalabilitas UI di masa depan. | Gunakan pola Singleton untuk model AI agar hemat RAM. | Deskripsikan alur pencarian "Google-style" untuk perankingan relevansi. |
| **Massive Testing** | Memastikan reliabilitas pada ratusan kueri. | Implementasi 201 test case menggunakan `pytest`. | 206/206 Lolos. | Kueri semantik sangat tangguh menangani variasi bahasa manusia. | Pengujian skala besar pada database remote rentan terhadap *network jitter*. | Minta AI membuat daftar kueri yang variatif untuk cakupan tes yang luas. |
| **Performance Audit** | Belum diketahui kebutuhan resource riil. | Benchmarking latensi & RAM usage via `psutil`. | Latensi <1s, RAM ~325MB. | E5-Small sangat ramah untuk VPS resource rendah (RAM 1-2GB). | Latensi didominasi network I/O pada database remote. | Instruksikan pembuatan laporan formal untuk dokumentasi keputusan arsitektur. |
| **UI/UX Design** | Butuh antarmuka yang familiar dan bersih. | Replikasi Google Search Original menggunakan Stitch. | Desain "Pure Search" yang minimalis & mobile-friendly. | Antarmuka familiar (Google-style) meminimalkan kurva pembelajaran user. | Filosofi "Invisibility of UI" mengutamakan utilitas di atas dekorasi. | Gunakan instruksi "Strict Replica" untuk mengarahkan AI membuat desain konsisten. |
| **Social Share** | Perlunya fitur awareness tanpa mengganggu UX. | Server-side image rendering (1:1) dan text copy dengan opsi branding toggle. | Pengguna dapat share konten estetik dengan kontrol penuh privasi. | Memberikan opsi "Hide Branding" membangun kepercayaan pengguna terhadap aplikasi. | Pisahkan fitur interaktif (Share) dari alur pencarian cepat (Result Page). | Definisikan skema sharing dalam API Contract sebelum mendesain UI. |
| **Backend API (FastAPI)** | Implementasi kontrak API yang sudah didefinisikan. | FastAPI dengan integrasi Search Library dan Image Generator. | API fungsional sesuai standar v1.1 divalidasi dengan Postman. | Postman Collection mempermudah integrasi frontend di masa depan. | Gunakan Response class (FastAPI) untuk mengirimkan data biner (gambar). | Instruksikan pembuatan Postman Collection sebagai langkah penutup fase backend. |
