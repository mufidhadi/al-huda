# Development Log: Tanya Quran Hadist

**Project Name:** Tanya Quran Hadist  
**Date:** 25 Maret 2026  
**Developer:** Gemini CLI & Mas Mufid  

## 🚀 Project Overview
Membangun sistem kueri dan dataset terpadu untuk teks suci Al-Quran dan Hadits. Fokus utama adalah pada integritas data, pemisahan struktural (Sanad vs Matan), dan optimasi performa penyerapan data menggunakan infrastruktur modern. Kini telah mendukung penyimpanan vektor untuk fitur Semantic Search berbasis AI.

## 🛠️ Tech Stack
- **Python 3.12**: Bahasa pemrograman inti.
- **UV**: Manajer paket dan proyek untuk eksekusi dependensi yang sangat cepat.
- **PostgreSQL (v16) + pgvector (v0.8.1)**: Basis data relasional dengan kemampuan Vector Database.
- **Psycopg2 (psycopg2-binary)**: *Driver* interaksi database.
- **Requests & HTTP Connection Pooling**: Pengambilan data dari API eksternal secara efisien.
- **Concurrent.futures (ThreadPoolExecutor)**: Paralelisme untuk memangkas waktu tunggu I/O.
- **tqdm**: Indikator progres visual untuk transparansi setiap proses.

## 📝 Experience & Task List

| Task | Issue | Tried Approach | Result | Lesson Learned |
| :--- | :--- | :--- | :--- | :--- |
| **Quran Ingestion** | Duplikasi data (11k ayat) karena proses paralel yang tumpang tindih. | Penggunaan `TRUNCATE` sederhana dan menjalankan skrip secara berulang di background. | Data terduplikasi karena proses background pertama belum mati saat proses kedua mulai. | Selalu bersihkan proses background (`taskkill`) sebelum menjalankan ulang ETL intensif. |
| **Gap Analysis & Resume** | Ingest data terputus di tengah jalan karena gangguan jaringan. | Menambahkan pengecekan jumlah ayat per surah sebelum menarik data API. | Skrip mampu melanjutkan (*resume*) tanpa menulis ulang data yang sudah ada. | Fitur *state-awareness* sangat krusial untuk dataset skala besar agar efisien waktu & kuota. |
| **Indexing Performance** | Pembuatan INDEX sangat lambat pada tabel yang sudah berisi ribuan baris. | Menambah `CREATE INDEX` setelah tabel terisi. | Mengalami `LockNotAvailable` karena tabel terkunci oleh koneksi liar. | Lakukan `VACUUM ANALYZE` dan tingkatkan `maintenance_work_mem` sebelum indexing. |
| **Hadith Master Data** | Struktur data hadits dari API seringkali tercampur-campur. | Membuat tabel master untuk Imam, Kitab, dan Derajat sebagai referensi. | Integritas referensial terjaga melalui *Foreign Key Constraints*. | Normalisasi data master di awal mempermudah kueri kompleks (Join) di kemudian hari. |
| **Sanad-Matan Split** | Tidak ada API yang memisahkan Rantai Perawi (Sanad) dan Isi Hadits (Matan). | Algoritma *Splitter* lokal menggunakan kata kunci transisi (e.g., "bersabda:", "berkata:"). | Data tersimpan secara atomik di kolom berbeda, memudahkan pencarian isi hadits. | *Domain-specific parsing* sederhana seringkali lebih efektif daripada menunggu API sempurna. |
| **Transactional Safety** | Duplikasi data hadits saat proses paralel gagal di tengah rentang (*range*). | Membungkus `DELETE` dan `INSERT` dalam satu blok transaksi `conn.commit()`. | Data tetap bersih; jika satu hadits gagal dalam satu range, seluruh range di-rollback otomatis. | **Atomisitas** adalah harga mati dalam sinkronisasi data paralel agar tidak terjadi sisa data korup. |
| **Parsing Logic V3** | Kasus khusus (ID 4, 5, 412, dsb) gagal dipisah karena delimiter terlalu awal atau ambigu. | Implementasi *Rightmost Search* (`rfind`) untuk mencari perawi terakhir dan penambahan delimiter tanda kutip (`berkata, "`). | Akurasi meningkat drastis (>95%) untuk dataset Bukhari. | Struktur hadits bersifat dinamis; pendekatan berbasis posisi (*positional*) lebih tangguh. |
| **Vector Support** | Perlunya dukungan untuk pencarian berbasis makna (Semantic Search). | Memeriksa ketersediaan dan mengaktifkan ekstensi `pgvector` di PostgreSQL. | Ekstensi `pgvector` v0.8.1 berhasil diaktifkan di database `tanya_quran_hadist`. | PostgreSQL kini siap menampung data *embedding* AI untuk fitur pencarian modern. |
